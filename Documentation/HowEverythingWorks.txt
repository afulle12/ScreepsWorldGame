# Complete System Integration Overview

## How Everything Fits Together

This Screeps AI is stored in the repository with `main.js` at the root and modules organized under `Bots/`, `Scanners/`, `Utilities/`, and `Documentation/`. `main.js` is the only tick entry point. It requires all managers, role modules, console-command modules, and side-effect modules, then runs them through a CPU-aware scheduler.

The current live build has two major consolidation points:

- `scanner.js` owns observer-backed intelligence, room registry, wide scans, player analysis, player monitoring, nuke analysis, nuke threat scans, maintenance scans, and war estimates.
- `repairManager.js` owns unified repair planning, nuke repair planning, rebuild caches, and repairer spawn requests. `roleRepairer.js` executes those plans.

## Main Loop Integration

`main.js` performs these high-level steps each tick:

1. Initialize cached room state through `getRoomState.init()`.
2. Refresh role counts and shared creep lookup tables.
3. Run critical defense, repair, and tower systems.
4. Run infrastructure managers: links, terminals, labs, factory, storage, room balance, mineral handling, power, and boost support.
5. Run market automation: pricing, market buy/sell workflows, auto trader, arbitrage, market lab, local refine, market refine, opportunistic buy/sell, and daily finance.
6. Run strategic systems: remote supply, claim route checks, tower drains, contested demolishers, nukes, deposit observation, and scanner workflows.
7. Run creeps by role, with role priority throttling under CPU pressure.
8. Run cleanup, profiling, memory maintenance, and status reporting.

## CPU Budget Model

The live build is tuned for Shard 3 with a 20 CPU limit.

`main.js` defines these budget assumptions:

- `HARD_CEILING`: 19.3 CPU, leaving room for Memory serialization.
- `NORMAL`: 18.0 CPU for normal operation.
- `CRITICAL`: 14.0 CPU when bucket is low.

Subsystems are assigned section tiers:

- `CRITICAL`: must run every tick, including creeps, defense, towers, tower fillers, and room-state initialization.
- `HIGH`: important infrastructure, including links, terminals, labs, boost manager, repair manager, spawn manager, factory, minerals, room balance, and auto energy buying.
- `NORMAL`: standard strategic work, including power, tower drains, contested demolishers, deposit observation, nuke launch support, claim range checks, and remote supply.
- `LOW`: deferrable work, including market scans, market refining, auto trader, daily finance, arbitrage, status reports, creep profiling, and CPU profiling.

Creeps also have role priorities. Important logistics, combat, defense, tower filling, maintainers, and static distributors are protected before low-priority economy or diagnostic roles.

## Core Data Flow

1. Game state enters through `Game.rooms`, `Game.creeps`, `Game.spawns`, `Game.market`, and `Memory`.
2. `getRoomState.js` builds cached room snapshots for structures, creeps, sources, minerals, and common lookups.
3. Managers read cached room state where possible instead of repeatedly calling expensive room searches.
4. Managers write intent state into their own `Memory` namespaces.
5. `spawnManager.js` reads role counts, room conditions, orders, and manager requests to spawn creeps.
6. Creep role modules execute per-creep behavior and update lightweight creep memory.
7. Profilers and status modules summarize runtime behavior for console inspection.

## Unified Scanner

`Scanners/scanner.js` replaces the older split scanner/intel modules in the current build. It consolidates these responsibilities:

- Maintenance scanner.
- Room intel and scoring.
- Cached room efficiency profiles.
- Observer scheduler.
- Room ownership registry.
- Wide scan and player discovery.
- Player analysis and creep census.
- Player monitoring and alerting.
- Nuke strike analysis.
- Incoming nuke detection.
- Hostile nuke threat scans.
- War estimate pipeline.

All observer use flows through one scheduler so systems do not fight over observers in the same tick. External modules can request observation through:

```javascript
require('scanner').observe.request(roomName, 'sourceName', priority)
```

Observer priority is roughly:

- One-shot console commands first.
- WAR-status hot polls.
- Deposit observer route/deposit scans.
- ENEMY/NEUTRAL/ALLY monitor hot polls.
- Registry sweep with leftover observer capacity.

Important scanner memory keys include:

- `Memory.maintScan`
- `Memory.roomEffProfile`
- `Memory.roomEffCache`
- `Memory.roomIntelPending`
- `Memory.nukeAnalyzePending`

## Unified Repair System

`repairManager.js` is the planning layer. It computes compact per-room plans and emits repairer spawn requests. `roleRepairer.js` is the generic executor that carries out those tasks.

The repair manager handles:

- Road and container repair priorities.
- Wall and rampart targets.
- Critical rampart emergency work.
- Tower repair queue limits.
- Wartime and peacetime repair tiers.
- Nuke landing damage plans.
- Rebuild caches for important structures.
- Inaccessible target tracking.
- Repairer body sizing and spawn request generation.
- Repair memory compaction.
- Pause/resume controls.

`spawnManager.js` reads `Memory.repairSpawnRequests` and spawns creeps with role `repairer`. Legacy wall repair, repair bot, rampart bot, and defense repair spawning is disabled/replaced by repair manager requests. Existing legacy creeps continue to run until they expire.

## Market And Economy Flow

The economy is split into several cooperating systems:

- `terminalManager.js`: terminal balancing, transfers, and operation processing.
- `storageManager.js`: resource reservations and storage coordination.
- `roomBalance.js`: intra-room distribution and balancing.
- `marketPricing.js`: weighted market pricing helpers.
- `autoTrader.js`: periodic profit-seeking trade analysis.
- `marketArbitrage.js`: buy/sell spread exploitation with energy-cost accounting.
- `marketLab.js`: lab reaction profit workflows.
- `marketRefine.js`: market-backed factory refining loops.
- `localRefine.js`: local factory/lab refinement without market dependency.
- `marketBuy.js` and `marketSell.js`: direct order workflows and reservation integration.
- `opportunisticBuy.js` and `opportunisticSell.js`: tactical market opportunities.
- `autoEnergyBuyer.js`: automatic energy or battery purchasing.
- `dailyFinance.js`: credit and transaction tracking.

## Spawn Flow

`spawnManager.js` is the central spawn orchestrator. It spawns core economy roles, special mission roles, repairers, lab bots, remote suppliers, single-source room creeps, military roles, and power-support creeps.

Major spawn inputs include:

- Room state and energy capacity.
- Per-room role counts from `statusReport` / `main.js`.
- Mission orders in `Memory`.
- Repair requests in `Memory.repairSpawnRequests`.
- Remote supply demand.
- Lab, market, mineral, claim, combat, and tower-drain requests.

Repair-specific spawn flow:

1. `repairManager.run()` computes repair tasks.
2. It writes compact requests into `Memory.repairSpawnRequests[roomName]`.
3. `spawnManager.run()` calls its repairer spawn manager.
4. A creep is spawned with `role: 'repairer'`, `homeRoom`, `repairKind`, and a task payload.
5. `roleRepairer.run(creep)` executes the assigned task.

## Creep Role Flow

`main.js` dispatches creeps by `creep.memory.role`. Each role module owns its behavior.

Important role groups:

- Economy: `harvester`, `supplier`, `staticDistributor`, `upgrader`, `builder`, `maintainer`.
- Repair and construction: `repairer`, `wallRepair`, `repairBot`, `rampartBot`, `remoteBuilder`, `roadBuilder`.
- Processing: `extractor`, `extractorAssistant`, `mineralCollector`, `labBot`, `powerBot`, `operator`.
- Remote and expansion: `claimbot`, `remoteSupplier`, `remoteBuilder`, `scout`, `signbot`.
- Combat: `defender`, `attacker`, `demolition`, `contestedDemolisher`, `towerDrain`, `towerFiller`, `skAttacker`, `controllerAttacker`, `comboBot`, `hd`, `squad`, `thief`, `nukeFill`.

## Documentation-Only Context

`Documentation/llmcontext.js` is pure documentation for LLM-assisted code review and generation. It has no runtime purpose and is safe if accidentally required.

## Integration Notes

- Prefer `getRoomState` for repeated room lookups.
- New observer consumers should use `require('scanner').observe.request(...)` rather than directly calling observers.
- New repair behavior should flow through `repairManager` and `roleRepairer` instead of re-enabling legacy repair spawns.
- New high-frequency systems should be added to the `SECTION_TIERS` map in `main.js`.
- New console globals should live in the module that owns the feature, not in `main.js`.
- Keep Memory compact. Derived values should usually be recomputed from game state rather than persisted.
